# Development

This guide covers the local development workflow for `@ahnv/n8n-nodes-imagekit`.

## Requirements

- Node.js compatible with the project dependencies
- npm
- Git
- The `vendor/sdk-generation` submodule checked out when regenerating from ImageKit's OpenAPI spec

## Install

```sh
npm install
```

## Common commands

```sh
npm run generate        # regenerate OpenAPI-backed resource descriptions
npm run generate:check  # verify generated resources are committed
npm run lint            # run n8n community-node lint rules
npm test                # run Jest tests
npm run build           # clean and build dist/
npm run dev             # run the node in the n8n node-cli dev harness
```

## Code generation

`npm run generate` reads:

- `vendor/sdk-generation/openapi/v1.0.0.yaml`
- `vendor/sdk-generation/stainless-config/main.yaml`
- `openapi/config.json`

It writes resource files under:

- `nodes/Imagekit/resources/_generated/`

`ImagekitTrigger.node.ts` is intentionally manually maintained and is not regenerated by `scripts/generate.ts`.

After changing `openapi/config.json` or updating the vendored SDK-generation submodule, run:

```sh
npm run generate
npm run generate:check
```

The generator may print drift warnings for OpenAPI or Stainless operations that are not mapped yet. Treat these as review prompts unless the script exits non-zero.

## Trigger node

`nodes/Imagekit/ImagekitTrigger.node.ts` is a handwritten trigger node. Keep these invariants in mind:

- It should stay next to `Imagekit.node.ts` so n8n groups Actions and Triggers under the same ImageKit panel.
- It should use the same icon paths as the main node.
- The `imagekitWebhookApi` credential must remain available on the node and reference `testedBy: 'imagekitWebhookSecretTest'`.
- Signature verification should use `verifyImagekitSignature` and raw request body data.

## Tests

Run all tests with:

```sh
npm test
```

Tests live in `__tests__/` and cover resources, uploads, trigger filtering, and webhook signature verification.

## Linting

Run:

```sh
npm run lint
```

The n8n community-node lint rules are strict and are focused on shipped runtime code. `scripts/**` and `__tests__/**` are excluded in `eslint.config.mjs` because codegen and tests intentionally use Node.js APIs that n8n Cloud runtime nodes should not use.

## Local n8n development

Run:

```sh
npm run dev
```

The build step cleans `dist/` first to avoid stale compiled node files being loaded by n8n.

If the dev harness fails with an npm cache error like `ENOTEMPTY` under `~/.npm/_npx`, remove the stale npx cache directory and retry:

```sh
rm -rf ~/.npm/_npx/<cache-id>
rm -rf ~/.n8n-node-cli
npm cache verify
npm run dev
```

## Release process

Releases are managed with `release-it` and conventional changelog generation.

```sh
npm run release
```

This updates the version, updates `CHANGELOG.md`, creates a git tag, and creates a GitHub release. npm publishing is handled by the GitHub Actions publish workflow when a version tag is pushed.

The npm package uses OIDC Trusted Publishing. The npm Trusted Publisher entry must match the workflow filename exactly:

- Repository: `ahnv/n8n-nodes-imagekit`
- Workflow: `publish.yml`
- Environment: blank

## Before pushing

Run:

```sh
npm run generate:check
npm run lint
npm test
npm run build
```

## Commit messages

Use conventional commits, for example:

```sh
fix(trigger): keep webhook credential available
feat(file): add new operation
chore(codegen): refresh generated resources
```

Commit linting is configured through Husky and commitlint.
